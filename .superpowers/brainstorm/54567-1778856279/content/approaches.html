Three Architecture Approaches

Pick the one that fits best — or tell me what to mix

A

Single Landing Zone + Streaming Fan-out

A generator job writes all synthetic events to one landing.synthetic_events Delta table. A single Structured Streaming job reads it, parses each event by type, and routes to the correct MVM target tables via foreachBatch.

Pros

  • One job to monitor and debug
  • Simple deployment (2 components)
  • Easy to replay / reprocess events

Cons

  • Fan-out routing logic grows complex with 153 tables
  • Single bottleneck for all domains
  • No native DLT data quality enforcement
Generator Job → landing.synthetic_events → Streaming Job (foreachBatch) → MVM Tables (all domains)
B

Domain-Partitioned Sources + Delta Live Tables ⭐ Recommended

The generator writes to domain-specific staging Delta tables (order_events, inventory_events, etc.). A Spark Declarative Pipeline (DLT) defines each MVM table as a @dlt.table with readStream, expectations, and FK enrichment. Deployed via Databricks Asset Bundle.

Pros

  • DLT handles checkpointing, retries, and lineage
  • Built-in data quality expectations per table
  • Scales per-domain independently
  • Full DAB deployment (bundle.yml declares pipeline + jobs)
  • Clean separation: generate → stage → transform → serve

Cons

  • More moving parts (staging tables + DLT pipeline)
  • DLT adds pipeline compute cost
Generator Job → Staging Tables (per domain) → DLT Pipeline (readStream per table) → MVM Silver Tables → Gold Metrics Views
C

Pure DLT with Custom Rate Source

The entire system lives inside one DLT pipeline. A custom Python function generates events inline using a @dlt.view with a rate-based trigger. No external generator job — DLT manages the full lifecycle from event creation to serving.

Pros

  • Minimal infrastructure — one pipeline does everything
  • Unified lineage from generation to gold layer
  • No separate generator job to schedule

Cons

  • Hard to backfill history (DLT rate source doesn't support time travel)
  • Harder to inject external API data (weather, events) into DLT source
  • Less control over event timing and realistic entropy
  • Can't easily replay or reseed data
DLT Pipeline → @dlt.view (inline generator) → MVM Tables
My recommendation

Option B — Domain-partitioned staging + DLT pipeline. The configurable backfill window, weather API injection, and 153-table MVM schema all need a generator job that runs independently from the streaming layer. DLT gives you quality enforcement and lineage for free. DAB deploys the whole thing in one command.
